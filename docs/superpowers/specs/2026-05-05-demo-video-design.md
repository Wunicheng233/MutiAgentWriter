# StoryForge AI 产品 Demo 视频设计规范

> **状态**: 待确认  
> **版本**: v1.0  
> **创建日期**: 2026-05-05  
> **适用范围**: 竞赛展示 / 产品宣传 Demo / 技术亮点展示

---

## 一、设计目标

制作一支 **90 秒左右、16:9、1080p** 的精致产品 Demo 视频，用“真实产品画面 + Remotion 动画包装”的混合路线，展示 StoryForge AI 从创作意图到可阅读作品的完整闭环。

视频定位为：**偏产品宣传，但兼顾竞赛评分**。它应当让评委和观众快速感知：

1. StoryForge AI 解决的是“AI 小说生成难以稳定、难以共创、难以风格化、难以交付”的问题。
2. AI 不是简单写一段文本，而是通过多智能体完成策划、写作、评审、修订、确认、阅读、导出。
3. 作家风格 Skill 注入层、逐章共创、质量门控是项目的差异化亮点。
4. 系统已经形成可演示、可落地、可继续扩展的产品形态。

---

## 二、核心风格原则

| 原则 | 要求 |
|------|------|
| **少解释** | 不做长段说明，不堆概念。每屏最多保留一句短文案。 |
| **多展示** | 以真实 UI、状态变化、章节生成、阅读器质感、导出流程为主。 |
| **有书香气** | 维持项目前端的米白、墨褐、松绿色、纸张感、宋体标题气质。 |
| **产品感** | 镜头推进、局部聚焦、状态标签、流程线和微动效要自然顺滑。 |
| **可信** | 核心功能画面优先来自真实项目截图或短录屏，避免纯概念动画。 |
| **克制** | 不使用夸张粒子、霓虹、过度渐变、娱乐化音效。 |

---

## 三、制作路线

采用 **Route C：真实产品画面 + Remotion 动画包装**。

### 3.1 真实素材层

来自本地项目的真实 UI 截图或短录屏：

- 书架页
- 新建项目流程
- 作家风格 Skill 选择
- 项目概览 / 大纲确认
- 逐章共创确认弹窗
- 编辑器
- 阅读器
- 质量分析
- 导出分享

### 3.2 Remotion 包装层

用于控制成片质感：

- 镜头推拉、平移、局部放大
- 页面之间的遮罩转场
- 文案淡入淡出
- 流程线和状态标签动画
- 光标/焦点提示
- 背景纸张纹理和柔和阴影
- 结尾 Logo / 价值总结

## 四、90 秒分镜设计

| 时间 | 画面 | 文案 | 功能点 |
|------|------|------|--------|
| 0-6s | 米白纸张质感背景，StoryForge AI 标题浮现，镜头缓慢推进到书架 UI | `让 AI 写小说，不止生成一段文字。` | 产品定位 |
| 6-14s | 书架页展示已有项目卡片，点击新建项目，页面流畅切换 | `从一个想法开始。` | 书架 / 新建入口 |
| 14-24s | 新建项目流程：内容类型、小说名、简介、协作方式，重点扫过作家风格选择 | `选择题材、协作方式和作家风格。` | 创建流程 / Skill 入口 |
| 24-34s | 作家 Skill 卡片聚焦，显示刘慈欣 / 鲁迅 / 海明威等风格选择，选中状态高亮 | `风格成为可注入的创作能力。` | 作家风格 Skill 注入 |
| 34-46s | 项目概览进入策划确认，AI 生成大纲，确认弹窗浮现 | `先策划，再写作。方向由人确认。` | 策划确认模式 |
| 46-60s | 逐章共创流程：章节生成中、评审、修订、等待确认，以流程线或状态卡表现 | `每一章，都经过写作、评审与修订。` | 多智能体工作流 |
| 60-70s | 章节确认弹窗展示，通过后自动进入下一章，进度整数增长 | `人在关键节点把关。` | Human-in-the-loop |
| 70-78s | 阅读器沉浸展示，分页翻页，文字排版和纸张空间作为视觉高潮 | `最终回到阅读本身。` | 阅读器 / 小说成品 |
| 78-84s | 质量分析页和字数/评分/问题定位短暂扫过 | `质量可检查，过程可追踪。` | 质量评分 / 过程可视化 |
| 84-90s | 导出分享页，最后回到产品 Logo 与一句收束 | `StoryForge AI：多智能体小说创作工作台。` | 导出 / 交付闭环 |

## 五、功能展示优先级

### 5.1 必须展示

1. 新建项目与核心配置
2. 作家风格 Skill 选择
3. 策划确认
4. 逐章共创确认
5. 多智能体写作流程状态
6. 阅读器成品效果
7. 质量分析
8. 导出分享

### 5.2 可以弱化

1. 设置页
2. 协作者管理
3. AI Assistant 侧栏
4. 详细过程日志
5. 过长的表单字段解释

### 5.3 不建议展示

1. 后端终端日志
2. GitHub Actions 页面
3. 大段技术架构图
4. 冗长评分标准文字

这些内容会削弱产品宣传片节奏。如需体现工程可靠性，应以“质量可检查，过程可追踪”一屏带过。

## 六、视觉系统

### 6.1 色彩

延续前端默认主题：

| 用途 | 色彩方向 |
|------|----------|
| 背景 | 米白、暖纸色、浅雾绿 |
| 文字 | 墨褐、暖灰褐 |
| 强调 | 松绿色 |
| 辅助强调 | 柔和橙色，仅用于生成中状态 |
| 阴影 | 低透明暖灰阴影 |

避免：

- 大面积深蓝/紫色科技风
- 纯白高亮导致纸张感消失
- 高饱和红绿对撞
- 复杂渐变背景

### 6.2 字体

| 场景 | 字体方向 |
|------|----------|
| 标题 | 宋体 / Serif，保持书卷感 |
| UI 文案 | 系统 sans-serif，保持清晰 |
| 数字 / 状态 | Inter / system-ui |

### 6.3 动效

| 动效 | 使用方式 |
|------|----------|
| 镜头推进 | 用于进入产品 UI 和阅读器高潮 |
| 局部放大 | 用于 Skill、确认弹窗、质量评分 |
| 遮罩转场 | 用于页面切换，模拟书页翻动或纸张滑入 |
| 进度变化 | 只显示整数百分比，避免浮点数 |
| 光标提示 | 仅在必要点击处短暂出现 |

## 七、文案策略

视频文案要像产品短片，不像说明书。

### 7.1 文案原则

- 每句不超过 18 个汉字为佳。
- 不解释实现细节，只命名价值。
- 不出现“本系统可以”“用户需要”“我们采用了”等说明句式。
- 关键创新点用短句落锚。

### 7.2 候选文案

1. `让 AI 写小说，不止生成一段文字。`
2. `从一个想法开始。`
3. `风格成为可注入的创作能力。`
4. `先策划，再写作。`
5. `人在关键节点把关。`
6. `每一章，都经过写作、评审与修订。`
7. `最终回到阅读本身。`
8. `质量可检查，过程可追踪。`
9. `多智能体小说创作工作台。`

## 八、素材需求

### 8.1 最低素材清单

| 素材 | 来源 | 说明 |
|------|------|------|
| 书架页截图/录屏 | 本地前端 | 展示产品入口 |
| 创建流程截图/录屏 | 本地前端 | 展示配置与 Skill |
| Skill 选择截图 | 本地前端 | 展示风格注入 |
| 策划确认截图/录屏 | 本地前端 | 展示人机共创 |
| 逐章确认截图/录屏 | 本地前端 | 展示逐章模式 |
| 阅读器录屏 | 本地前端 | 展示最终作品 |
| 质量分析截图 | 本地前端 | 展示可控性 |
| 导出页截图 | 本地前端 | 展示交付闭环 |

### 8.2 素材采集要求

1. 使用稳定演示项目，不在录制时依赖真实长任务生成。
2. 必要时使用已生成好的项目数据，保证画面可复现。
3. 截图统一 16:9 或可安全裁切到 16:9。
4. 避免录入浏览器地址栏、系统通知、调试窗口。
5. UI 状态应真实，不能出现卡死、空数据、异常标签。

## 九、Remotion 实现方案

### 9.1 项目位置

建议新建：

```

```

或在不污染主前端工程的情况下放入：


推荐使用 `tools/demo-video/`，因为视频制作是展示工具，不属于主产品运行时代码。

## 十、风险与规避

| 风险 | 规避方式 |
|------|----------|
| 真实流程生成耗时过长 | 使用预生成演示项目和截图/短录屏 |
| UI 画面状态不一致 | 录制前固定演示数据，逐页检查状态 |
| 视频文字过多 | 每屏只保留一句短文案 |
| 技术亮点讲不清 | 通过流程状态和确认弹窗表达，不用架构图堆概念 |
| 画面像 PPT | 使用真实 UI 镜头运动和局部细节，不做纯静态卡片 |
| Remotion 影响主项目 | 独立放在 `tools/demo-video/`，不接入前端构建链 |

---

## 十一、验收标准

1. 成片控制在 85-95 秒之间。
2. 画面比例为 16:9，分辨率 1920x1080。
3. 必须完整展示创建、Skill、策划确认、逐章共创、阅读器、质量分析、导出。
4. 文案简洁，不出现大段说明。
5. 风格与当前前端一致：米白、墨褐、松绿、书香气。
6. 不出现空项目、报错、浮点进度、无内容章节等异常状态。
7. 可用一条命令重新渲染成片。

---

## 十二、待确认事项

1. 是否接受本设计中的 90 秒分镜顺序。
2. 是否接受 `tools/demo-video/` 作为 Remotion 工程位置。
3. 是否需要配音。默认建议第一版不做配音，使用背景音乐和字幕即可。
4. 是否需要英文字幕。默认建议中文优先，最后可补英文版。
