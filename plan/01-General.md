## 🔍 一、现状诊断：核心痛点与缺失能力

基于项目结构和 README 描述，我识别出以下**阻碍产品化**的关键问题：

### 1. 架构层面的不足
| 问题 | 影响 |
|------|------|
| **单体流水线耦合过重** | `main.py` 包含所有流程逻辑，新增内容类型或修改流程需要改动大量代码。 |
| **Agent 实例化重复开销** | 每次生成章节都重新创建 Agent 对象，浪费连接池缓存优势。 |
| **状态管理原始** | 世界观状态依赖 JSON 文件读写，无事务性，断电或异常易损坏。 |
| **缺少任务队列** | 生成是同步阻塞的，用户必须等待完整流程结束，无法后台运行或暂停恢复。 |
| **无用户系统** | 所有作品存在本地文件夹，无法多用户隔离，也不能云端部署。 |
| **配置管理混乱** | API 密钥、模型参数散落在环境变量、config.py、YAML 中，缺少统一配置中心。 |

### 2. 功能体验的缺失
| 缺失功能 | 用户痛点 |
|----------|----------|
| **生成过程不可干预** | 用户不能在章节生成后手动修改再继续，缺乏“人机协作”感。 |
| **无版本/历史记录** | 修改后无法回滚到之前的版本，实验性调整风险高。 |
| **导出格式单一** | 仅支持静态 HTML，无法导出 .epub、.docx 等通用格式。 |
| **无协作/分享机制** | 作品只能自己看，无法邀请他人评阅或协作。 |
| **缺少使用分析** | 没有 Token 消耗统计、成本预估，用户不知道花了多少钱。 |

### 3. 前端 UI 的现状
- 当前使用 Flask + 原生 HTML/CSS（模板在 `templates/`），风格朴素，缺乏统一设计语言。
- 移动端响应式已考虑，但整体交互停留在“工具感”，距离“专业写作产品”有较大差距。
- 没有使用现代前端框架，动态交互（如实时进度、章节内编辑）实现困难。

## 🎯 二、产品化目标定义

将 StoryForge AI 从一个**功能验证型项目**升级为**可部署、可运营、体验专业的 SaaS 级写作工具**。

**核心产品定位**：
> 为严肃长文创作者（网文作者、剧本编剧、内容工作室）提供的 **AI 辅助写作工作台**，强调一致性保障、质量可控、人机协作。

**关键成功指标**：
1. 用户 5 分钟内可完成首个章节生成。
2. 长篇生成过程中，一致性错误率 < 5%。
3. 生成内容可直接导出为行业标准格式。
4. 支持 100+ 并发用户（轻量部署）。

### 3.2 重构优先级与步骤

| 阶段 | 任务 | 产出 | 预估工时 |
|------|------|------|----------|
| **Phase 1：基础解耦** | 1. 将现有 `main.py` 拆分为 `Orchestrator` 类 <br> 2. Agent 实例化改为单例池 <br> 3. 引入 Pydantic Settings 管理配置 | 可独立运行的 Core 模块，仍可命令行调用 | 2-3 天 |
| **Phase 2：引入任务队列** | 1. 集成 Celery + Redis <br> 2. 将生成流程封装为 Celery 任务 <br> 3. 实现进度回调（WebSocket 或 Server-Sent Events） | 支持后台生成，前端可获取实时进度 | 3-4 天 |
| **Phase 3：API 层与数据库** | 1. 使用 FastAPI 构建 RESTful API <br> 2. 设计 PostgreSQL 数据模型（User, Project, Chapter, AgentLog） <br> 3. 实现用户认证（简单的 API Key 或 JWT） | 完整的后端 API，支持多用户 | 4-5 天 |
| **Phase 4：前端现代化** | 1. 使用 React + TypeScript + Tailwind CSS 重写前端 <br> 2. 集成 `DESIGN.md` 定义的书籍风格设计系统 <br> 3. 实现核心页面：仪表盘、创作向导、章节编辑器、质量分析 | 专业级前端应用 | 7-10 天 |
| **Phase 5：高级功能** | 1. 章节版本管理 <br> 2. 导出服务（.epub, .docx） <br> 3. 使用量统计与成本预估 <br> 4. 协作分享（生成只读链接） | 完整 SaaS 功能集 | 5-7 天 |

## 🎨 四、前端产品规划（基于上一轮的 DESIGN.md）

我们已有一份精心设计的**书香气设计系统**，现在需要将它落地为具体的页面和组件。

### 4.1 页面结构

```
/ (首页) → 如果未登录重定向到 /login
├── /dashboard          # 我的书架（项目列表）
├── /create             # 创作向导（4步）
├── /project/:id        # 项目详情页
│   ├── /overview       # 概览：大纲、人设、进度
│   ├── /chapters       # 章节列表 + 逐章阅读
│   ├── /write/:chapId  # 章节编辑器（分屏：左侧智能体面板，右侧写作区）
│   └── /analytics      # 质量仪表盘
├── /settings           # 设置（API Key、偏好）
└── /share/:token       # 分享只读页面
```

### 4.2 关键交互设计（与 DESIGN.md 结合）

| 页面 | 设计要点 |
|------|----------|
| **Dashboard** | 卡片式项目陈列，使用 Vellum 白色半透明卡片，软阴影。每个卡片显示封面（或占位图）、标题、进度条、质量评分徽章。 |
| **创作向导** | 4 步表单，每步卡片化，Sage Green 按钮。进度条使用 Terracotta 色。 |
| **写作编辑器** | 左右分栏。左栏纵向排列 8 个 Agent 卡片（可折叠），实时显示状态（思考中、完成、警告）。右栏是 720px 固定宽度的写作画布，Crimson Pro 字体，背景 Parchment。底部工具栏包含“请求建议”、“检查一致性”、“保存”等操作。 |
| **质量仪表盘** | 使用 ECharts 绘制雷达图和趋势图，配色取自系统（Sage、Terracotta、Faded Rose）。逐章问题列表以对话气泡形式展示 Critic 的评语。 |

### 4.3 技术选型建议

- **框架**：React 18 + TypeScript
- **样式**：Tailwind CSS（自定义主题映射 DESIGN.md 的色板）
- **状态管理**：Zustand（轻量）或 TanStack Query（服务端状态）
- **路由**：React Router v6
- **图表**：ECharts
- **富文本编辑**：TipTap（基于 ProseMirror，易扩展 Markdown 支持）
- **实时通信**：Socket.IO 或原生 WebSocket

## 🛠️ 五、Claude Code 可执行的任务拆分

为了让 Claude Code 高效协助，应将上述规划拆分为**独立、可验证、有明确输入输出**的任务。以下是一份可直接喂给 AI 的任务清单示例：

### Task 1: 核心模块解耦与配置管理
- **输入**：现有 `main.py`, `agents/`, `config.py`
- **要求**：
  1. 创建 `core/orchestrator.py`，包含 `NovelOrchestrator` 类，将 `main.py` 中的生成流程迁移进去，保留命令行入口。
  2. 创建 `core/agent_pool.py`，实现 Agent 单例池（根据 agent_name 缓存实例）。
  3. 使用 Pydantic Settings 重构配置，支持从 `.env` 和 YAML 统一加载。
- **验收**：运行 `python -m core.orchestrator --project-name test` 能正常生成。

### Task 2: 引入 Celery 任务队列
- **输入**：`core/orchestrator.py`
- **要求**：
  1. 安装 Celery、Redis，编写 `celery_app.py`。
  2. 创建 `tasks/writing_tasks.py`，定义 `generate_novel_task`，内部调用 Orchestrator。
  3. 实现进度更新机制（例如通过 Redis pub/sub 或 WebSocket 回调）。
- **验收**：通过 Celery worker 异步启动生成，并能从 Redis 获取进度百分比。

### Task 3: FastAPI 后端搭建与数据库模型
- **输入**：现有数据模型（隐含在 YAML 结构中）
- **要求**：
  1. 使用 SQLAlchemy 定义 User, Project, Chapter, AgentLog 表。
  2. 实现用户注册/登录 API（简单 API Key 模式即可）。
  3. 实现项目管理 CRUD API。
  4. 实现“创建项目”接口，内部触发 Celery 任务。
- **验收**：可通过 Swagger UI 测试各接口。

### Task 4: 前端项目初始化与设计系统落地
- **输入**：上一轮提供的 `DESIGN.md`（书香气设计系统）
- **要求**：
  1. 使用 Vite 创建 React+TS 项目，配置 Tailwind。
  2. 在 `tailwind.config.js` 中扩展颜色、字体、阴影，完全映射 DESIGN.md。
  3. 创建基础组件：Button, Card, Badge, Input, Navigation。
  4. 实现 Dashboard 页面静态版本。
- **验收**：运行 `npm run dev` 能看到符合设计规范的 Dashboard。

### Task 5: 写作编辑器核心交互
- **输入**：项目详情 API
- **要求**：
  1. 实现 `/project/:id/write/:chapId` 页面布局（左右分栏）。
  2. 左栏渲染 8 个 Agent 卡片（静态数据），右栏集成 TipTap 编辑器。
  3. 实现章节内容的加载与自动保存（防抖）。
- **验收**：能够在编辑器中修改章节内容并保存到后端。

## 💡 七、其他建议

1. **成本控制**：建议在 Orchestrator 中加入 Token 预估和硬上限，避免用户误操作产生巨额账单。
2. **内容安全**：Compliance Agent 目前主要做合规检查，可考虑接入第三方内容审核 API 增强。
3. **开源与商业化**：如果计划开源，可保持 Core 模块 MIT 协议，将 SaaS 特有功能（用户系统、队列）作为 Pro 版本。
4. **文档**：建议在重构过程中同步编写开发者文档，方便后续贡献者。
