# StoryForge AI 长期执行总计划

## 1. 文档定位

这份文档是 StoryForge AI 当前阶段最重要的总纲文件。

它的目标不是重复愿景，也不是替代具体开发计划，而是把前面的两份文档：

- `03-Long-Term Product Roadmap.md`
- `04-Architecture Evolution Plan.md`

进一步压缩成一份真正可执行的长期推进总计划，让团队后面所有产品、架构、研发、评测、社区建设工作，都能围绕同一条主线推进。

这份文档回答五个问题：

1. 我们未来两年究竟在建设什么
2. 这些事情应该按什么顺序做
3. 每个阶段的核心目标、边界和验收标准是什么
4. 哪些项目必须先做，哪些项目现在不能做
5. 后续更细的计划应该从这份文档怎么拆

在执行层面，这份总计划还需要配合一份更偏技术落地的 companion 文档使用：

- `06-Architecture Program Board.md`

`05` 负责定义总阶段和主线判断，`06` 负责把最新架构风险、依赖顺序和下一波施工主题收敛成更适合研发推进的总表。

## 2. 总体判断

### 2.1 我们现在最该避免的错误

StoryForge AI 当前最大风险，不是功能不够多，而是推进顺序错了。

如果现在同时推进：

- UI 大改
- 多 Agent 复杂化
- 社区广场
- 商业化
- 推荐流

最后大概率会得到一个看起来很丰富、但创作内核不稳定、状态管理混乱、无法持续优化的系统。

所以接下来一段时间，我们必须坚持一个非常明确的总策略：

**先把 StoryForge AI 做成稳定的 AI 原生创作工作台，再把它扩展成小说平台，最后才做生态和商业化。**

### 2.2 总策略

整个项目分三层推进：

1. `创作内核`
   解决“能不能稳定产出一部可持续写下去的小说”

2. `发布闭环`
   解决“作品能不能被展示、被阅读、被反馈”

3. `平台生态`
   解决“创作者为什么长期留下来，读者为什么长期回来”

### 2.3 接下来 12 个月的真实主线

如果只允许用一句话概括未来 12 个月：

**以 workflow、artifact、harness、author workspace 为核心，完成从 demo 到可持续创作系统的跨越。**

也就是说，未来一年真正的主战场不是“多做几个 Agent”，而是下面四件事：

- `Workflow`
- `Artifact`
- `Harness`
- `Workspace`

## 3. 执行原则

### 3.1 原则一：先稳定主路径，再扩展边缘功能

每个阶段都先保证主路径可用：

`idea -> plan -> chapter generation -> review -> edit -> publish`

只有主路径稳定，周边功能才有意义。

### 3.2 原则二：所有长期能力都必须可度量

未来每一个重要方向都要绑定指标。

不能只说：

- “质量更好了”
- “一致性更强了”
- “社区更活跃了”

而要定义出能够持续观测的指标。

### 3.3 原则三：数据库保存事实，文件系统保存快照

后续所有新增能力都必须遵守这一点：

- 数据库是业务真相
- 文件是快照与导出
- 向量索引是派生层

### 3.4 原则四：平台能力要顺着创作能力长出来

不是先做一个“小说论坛”再把创作功能塞进去，而是先把创作工作台做好，再让作品发布、评论、追更、推荐、打赏顺势长出来。

### 3.5 原则五：每个阶段都必须有明确的不做事项

长期项目最容易被“听起来很合理”的功能打散。

所以每个阶段不仅要定义做什么，更要定义：

**这个阶段坚决不做什么。**

## 4. 总体阶段拆分

建议将未来两年的推进拆成 6 个阶段。

### 阶段 0：基础收口期

时间建议：

- 第 0 到 1 个月

核心目标：

- 收口当前 demo 的认知与架构分叉
- 稳住创作主路径
- 建立后续重构和扩展的统一语言

### 阶段 1：创作内核稳定期

时间建议：

- 第 1 到 3 个月

核心目标：

- 把当前创作系统从“线性脚本 + 文件驱动”升级到“可恢复、可版本化、可解释的创作底座”

### 阶段 2：工作流与评测成型期

时间建议：

- 第 3 到 6 个月

核心目标：

- 建立 workflow runtime v1
- 建立最小可用 harness
- 让创作质量优化进入有证据的工程循环

### 阶段 3：作者工作台成型期

时间建议：

- 第 6 到 9 个月

核心目标：

- 从“生成器”升级成“创作者持续使用的工作台”

### 阶段 4：发布与社区闭环期

时间建议：

- 第 9 到 15 个月

核心目标：

- 作品可发布、可被发现、可被评论、可被追更

### 阶段 5：生态与商业化验证期

时间建议：

- 第 15 到 24 个月

核心目标：

- 验证平台留存与创作者收益模型

## 5. 阶段 0：基础收口期

### 5.1 阶段目标

这个阶段不是“做很多新功能”，而是把当前项目从“功能已存在但认知不稳定”的状态，整理为“所有人都知道当前系统是什么、未来往哪走”的状态。

### 5.2 要完成的事情

#### A. 文档与现实收口

- 明确当前真实创作主路径，以当前 `orchestrator` 为准
- 标记已有文档中与当前运行架构不一致的部分
- 形成统一术语表：
  - project
  - chapter
  - workflow run
  - artifact
  - feedback
  - revision
  - publish

#### B. 主路径稳定性清理

- 修复已知高优先级 bug
- 为创作、恢复、反馈、编辑、导出、分享补足回归测试
- 对当前异步生成链路补最小化错误追踪

#### C. 架构边界初步梳理

- 明确哪些状态现在还在文件里
- 明确哪些状态已经在数据库里
- 标出所有“单一事实源不明确”的点

#### D. 项目管理方式建立

- 建立统一阶段计划文档体系
- 建立决策记录机制
- 建立阶段验收与复盘机制

### 5.3 本阶段交付物

- 一份稳定的总计划主文档
- 一份术语表或系统字典
- 一份当前系统状态图
- 一套核心链路回归测试
- 一份“当前状态源梳理清单”

### 5.4 验收标准

- 团队内部对“系统当前怎么跑”没有分歧
- 核心流程 bug 有回归测试覆盖
- 后续每一个重构任务都能映射到统一架构语言

### 5.5 本阶段不做

- 不做社区广场
- 不做推荐流
- 不做打赏
- 不做复杂多分支创作 UI
- 不做新的复杂 Agent 角色扩充

## 6. 阶段 1：创作内核稳定期

### 6.1 阶段目标

这是最关键的第一阶段。

目标不是把系统做得更“炫”，而是把它做成：

- 可恢复
- 可版本化
- 可解释
- 可控

### 6.2 本阶段核心命题

要解决四个根问题：

1. 当前一次创作运行到底如何被定义
2. 当前章节的最新内容、历史内容、反馈内容到底谁是主真相
3. 用户修改意见如何进入系统，而不是只存在文本文件里
4. 系统如何知道自己应该从哪里恢复继续跑

### 6.3 产品侧重点

- 用户能看到当前任务处于哪个真实状态
- 用户能区分“草稿”“待确认”“确认通过”“重写中”“发布版”
- 用户的反馈会被系统明确记录，而不是隐式吃掉
- 用户可以看到章节历史版本与恢复点

### 6.4 技术侧重点

#### A. 建立创作运行模型

至少引入以下概念：

- `run`
- `step`
- `artifact`
- `feedback`
- `revision`

#### B. 反馈结构化

把当前的 `feedback_x.txt` 机制逐步迁移为结构化记录，至少区分：

- 计划级反馈
- 章节级反馈
- 风格反馈
- 内容反馈
- 节奏反馈
- 人设反馈

#### C. 章节版本化

至少要支持：

- 当前版本
- 历史版本
- 用户手动修改版本
- AI 修订版本
- 可恢复到指定版本

#### D. 任务恢复显式化

让系统恢复依赖：

- 数据库状态
- checkpoint
- step 结果

而不是优先依赖：

- 文件是否存在

#### E. 检索隔离与上下文清理

逐步去掉依赖全局变量的检索上下文方式，使每次运行的上下文边界明确。

### 6.5 本阶段建议里程碑

#### 里程碑 M1：状态收束

- 明确项目、任务、章节、版本的关系
- 当前章节状态在数据库中可被完整追踪

#### 里程碑 M2：反馈入库

- 用户反馈结构化存储
- 反馈可以追溯到具体章节与版本

#### 里程碑 M3：最小恢复能力

- 任务失败后可以从最近稳定节点恢复
- 用户拒绝章节后可以明确重跑当前章而不是靠文件推断

### 6.6 阶段验收标准

- 核心创作链路不再依赖文件作为唯一状态判断依据
- 用户反馈不会丢失
- 章节历史版本和当前版本关系清晰
- 至少能稳定支持“重写当前章”“恢复某版本”“继续下一章”

### 6.7 本阶段不做

- 不做场景级多 Agent 拆解
- 不做复杂推荐系统
- 不做平台作者成长体系
- 不做商业化

## 7. 阶段 2：工作流与评测成型期

### 7.1 阶段目标

从这一阶段开始，StoryForge AI 要从“一个带状态的生成系统”进化成“一个有工作流引擎雏形、并且能做系统性优化的创作系统”。

### 7.2 本阶段核心命题

要解决两个长期根问题：

1. Workflow 如何显式化
2. 优化如何从拍脑袋变成可回归

### 7.3 产品侧重点

- 用户能看到创作链路当前走到了哪一步
- 用户能理解“为什么这章被打回”
- 用户能对某一步进行重试或回退

### 7.4 技术侧重点

#### A. Workflow Runtime v1

建立最小工作流运行时，支持：

- step 定义
- step 结果持久化
- step 重试
- approval 节点
- 条件分支
- checkpoint

#### B. Agent Contract v1

为核心 Agent 明确：

- 输入 schema
- 输出 schema
- prompt version
- model policy
- 错误类型

#### C. Harness v1

建立最小可用评测系统，至少包含：

- golden set
- failure set
- 固定运行配置
- 自动生成评测报告

#### D. 模型调用可观测

记录：

- 模型版本
- prompt 版本
- token 消耗
- 章节耗时
- step 成功率

### 7.5 本阶段建议里程碑

#### 里程碑 M4：Workflow Runtime 雏形

- 核心创作链路由 step 驱动
- 每一步都有标准化记录

#### 里程碑 M5：Harness 雏形

- 固定样本集可以回归跑通
- 输出稳定评测报告

#### 里程碑 M6：质量优化有证据

- 每次 prompt / workflow 调整都能比较前后差异

### 7.6 阶段验收标准

- 章节生成主路径已经不是单纯过程脚本
- 至少能做离线回归评测
- 团队可以回答“为什么这次质量更好或更差”

### 7.7 本阶段不做

- 不做大规模社区系统
- 不做复杂商业化逻辑
- 不做过早微服务化

---

## 8. 阶段 3：作者工作台成型期

### 8.1 阶段目标

到了这个阶段，产品形态要发生一次明显变化：

从“生成后让用户看结果”

变成：

“让作者在一个持续可控的工作台里创作、修订、对比、发布。”

### 8.2 本阶段核心命题

要解决的问题是：

- 作者为什么愿意连续几周都在这里创作
- AI 如何真正成为作者的副驾，而不是一次性代写器

### 8.3 产品侧重点

#### A. 灵感入口重构

把项目创建流程从“表单”升级为“灵感孵化向导”，至少支持：

- idea 输入
- 题材选择
- 目标平台
- 风格偏好
- 人物核心冲突
- 是否要 AI 扩写 premise

#### B. 作者工作台

作者工作台应至少包含：

- 项目总览
- 设定面板
- 大纲面板
- 章节状态面板
- 版本面板
- AI 建议面板
- 发布面板

#### C. 精细化修订能力

支持：

- 只重写这一章
- 只改开头
- 只改结尾
- 保留剧情重写文风
- 保留结构增强节奏
- 人设一致性修正

#### D. 创作解释能力

用户应能看到：

- 本章为什么被判不通过
- 本次修订重点是什么
- 本章与设定或大纲的主要冲突点在哪

### 8.4 技术侧重点

- artifact 与 workspace 联动
- 版本树与差异对比
- 结构化记忆初步落地：
  - 角色卡
  - 时间线
  - 伏笔表

### 8.5 本阶段建议里程碑

#### 里程碑 M7：作者工作台可用

- 用户能在一个统一界面完成创作、查看状态、发起修订

#### 里程碑 M8：局部修订可用

- 不需要每次重跑整章

#### 里程碑 M9：结构化记忆初步可用

- 系统开始真正理解故事状态，而不是只做向量召回

### 8.6 阶段验收标准

- 创作者可以持续使用系统推进一部长篇作品
- 人在环路不再只是“通过/不通过”
- 系统支持更细颗粒度的共创与修订

### 8.7 本阶段不做

- 不做复杂读者社区机制
- 不做大规模推荐流
- 不做收益分成系统

## 9. 阶段 4：发布与社区闭环期

### 9.1 阶段目标

从这一阶段开始，StoryForge AI 不再只是创作工具，而开始具备平台属性。

### 9.2 本阶段核心命题

要解决的问题是：

- 写出来的作品如何被展示
- 读者如何参与
- 作者如何从反馈中继续迭代

### 9.3 产品侧重点

#### A. 发布链路完整化

至少包含：

- 作品封面
- 简介
- 标签
- 连载页
- 章节页
- 作者主页
- 分享与传播

#### B. 作品广场

至少支持：

- 最新发布
- 热门作品
- 分类浏览
- 标签筛选
- 作者主页跳转

#### C. 读者互动

至少支持：

- 评论
- 点赞
- 收藏
- 关注作者
- 追更提醒

#### D. 评论反哺创作

把评论与作者反馈区分开，但逐步建立桥梁：

- 作者可将评论采纳为创作建议
- 评论可转成候选修订输入

### 9.4 技术侧重点

- 发布态与创作态分离
- 评论与审核机制
- 内容可见性控制
- 阅读侧分析数据

### 9.5 本阶段建议里程碑

#### 里程碑 M10：发布主链路闭环

- 创作者可以从创作直接进入发布

#### 里程碑 M11：作品广场 beta

- 平台内有真实可浏览内容池

#### 里程碑 M12：读者反馈闭环

- 评论、关注、追更开始形成作者留存动力

### 9.6 阶段验收标准

- 平台内存在连续更新的作品
- 作品不只是被生成，而是被阅读和反馈
- 作者和读者之间出现真实互动

### 9.7 本阶段不做

- 不急着做复杂推荐算法
- 不急着做大规模商业化活动体系
- 不急着做平台外多端扩张

---

## 10. 阶段 5：生态与商业化验证期

### 10.1 阶段目标

在创作内核、作者工作台和作品社区都初步稳定后，再进入生态验证。

### 10.2 本阶段核心命题

要回答三个问题：

1. 创作者能不能在这里获得持续激励
2. 读者是否愿意形成平台消费与互动习惯
3. 平台是否存在可持续商业模式

### 10.3 产品侧重点

#### A. 创作者激励

候选方向：

- 打赏
- 礼物
- 作品热度展示
- 作者成长体系
- 创作者数据面板

#### B. 平台商业化

候选方向：

- 高级创作套餐
- 高级模型额度
- 高级发布工具
- 创作者订阅

#### C. 生态扩展

候选方向：

- 共创模式
- 编辑协作
- 约稿与悬赏
- IP 延展工具

### 10.4 本阶段验收标准

- 创作者有明确长期留下的激励
- 读者有持续回访的理由
- 平台有至少一条被验证的收入路径

### 10.5 本阶段不做

- 不盲目扩展到过多内容品类
- 不在创作内核尚未稳定时强推变现

---

- 结构化记忆
- 局部修订能力
- 发布链路
- 作品广场 beta
- 评论与追更

### P2：可以后置

- 排行榜
- 推荐流
- 创作者成长体系
- 打赏
- 礼物
- 分成

### P3：更长期探索

- 共创网络
- 编辑市场
- IP 扩展工具
- 商业授权体系

## 12. 每个阶段都要有的标准输出

为了避免后面又回到“想到哪做到哪”，建议以后每个阶段都固定产出以下 6 类文档或结果。

### 12.1 阶段目标说明

说明本阶段只解决什么问题，不解决什么问题。

### 12.2 架构设计说明

说明本阶段新增的：

- 模块
- 数据结构
- 工作流节点
- 风险点

### 12.3 详细开发计划

拆到可执行任务级别。

### 12.4 验收指标

定义产品指标、技术指标、稳定性指标。

### 12.5 回归验证结果

尤其是创作主路径和 harness 结果。

### 12.6 阶段复盘

明确：

- 做对了什么
- 做错了什么
- 下一阶段是否需要调整顺序

## 13. 推荐的推进节奏

### 13.1 月度节奏

建议每个月都围绕一个主主题推进，而不是并行铺太多线。

建议节奏：

- 第 1 周：设计与拆解
- 第 2 周：核心实现
- 第 3 周：联调与回归
- 第 4 周：验收、复盘、下一阶段设计

### 13.2 双周节奏

建议两周一个小里程碑：

- 第一周完成骨架
- 第二周完成打磨与验证

### 13.3 版本节奏

建议每个大阶段形成一个明确版本主题，例如：

- `v0.2 创作底座版`
- `v0.3 Workflow 版`
- `v0.4 Harness 版`
- `v0.5 Author Workspace 版`
- `v0.6 Publish Beta 版`

## 15. 当前建议的下一步

如果完全按这份总计划执行，接下来最合理的落地顺序是：

1. 先继续补齐 `阶段 1` 的细化计划
2. 第一份要展开的详细计划应是：
   `创作运行模型与数据层改造计划`
3. 在它明确之后，再展开：
   `Workflow Runtime v1 落地计划`
4. 然后是：
   `Harness v1 建设计划`
5. 最后再进入：
   `Author Workspace v1 产品计划`

原因很简单：

- 没有数据层和运行模型，workflow 会虚
- 没有 workflow 和评测，workspace 会很脆
- 没有创作底座，社区功能会失去根

## 16. 一句话结论

StoryForge AI 的正确推进方式，不是“继续堆 demo 功能”，而是：

**先把创作底座做成系统，再把创作系统做成工作台，再把工作台做成平台。**

只要顺序不乱，后面你想要的：

- 多智能体创作
- 作者持续创作
- 作品发布广场
- 评论追更
- 打赏和生态

都会自然长出来，而且不会每走一步都推倒重来。
