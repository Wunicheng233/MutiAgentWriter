# Task 5: 产品增强与高级功能

## 一、任务目标

在核心创作流程稳定运行的基础上，补充让产品从“可用”迈向“优秀”的关键高级功能。这些功能直接提升用户粘性、协作体验和商业化潜力。

**核心产出**：
1. 多格式导出服务（`.epub`、`.docx`）
2. 章节版本历史与回滚
3. Token 用量统计与成本预估
4. 只读分享链接
5. 简单的协作功能（项目邀请）

## 二、功能分项规划

### 2.1 多格式导出服务

**用户价值**：让用户将生成的小说带走，用于投稿、自出版或本地备份。

**导出格式要求**：
| 格式 | 应用场景 | 技术方案 |
|------|----------|----------|
| `.epub` | 电子书阅读器（Kindle、Apple Books） | `EbookLib` 或 `pandoc` |
| `.docx` | Word 编辑、投稿 | `python-docx` |
| `.html` | 已有静态网站导出，保持即可 | 复用现有 `export_static.py` |

**实施要点**：
- 创建独立的 `ExportService` 类（`services/export_service.py`）。
- 接收 `project_id`，从数据库拉取章节内容和元数据。
- 生成临时文件，上传至对象存储（或本地静态目录），返回下载 URL。
- API 端点：`POST /api/projects/{id}/export?format=epub`，返回 `{download_url: "..."}`。
- **注意**：`.epub` 需正确处理封面图片、目录结构、元数据（作者、书名）。

### 2.2 章节版本历史

**用户价值**：允许用户在修改后回滚到之前的版本，消除“改坏了”的焦虑。

**数据模型扩展**：
- 新增表 `chapter_versions`：
  - `id`、`chapter_id`、`content`、`word_count`、`created_at`、`version_number`
- 每次通过编辑器保存章节时，自动将当前内容插入 `chapter_versions`。
- 保留最近 10 个版本（可配置），超出的自动清理。

**API 端点**：
| 方法 | 路径 | 功能 |
|------|------|------|
| GET | `/api/chapters/{id}/versions` | 获取版本列表（仅元数据） |
| GET | `/api/chapters/{id}/versions/{vid}` | 获取指定版本内容 |
| POST | `/api/chapters/{id}/restore/{vid}` | 恢复到指定版本 |

**前端交互**：
- 在编辑器工具栏增加“历史版本”按钮，点击弹出侧边抽屉，显示版本时间线。
- 点击版本可预览，确认后替换当前编辑器内容。

### 2.3 Token 用量统计与成本预估

**用户价值**：透明度——让用户知道每次生成花了多少钱，便于控制预算。

**数据采集点**：
- 在每个 Agent 调用 LLM 时，从 API 响应中提取 `usage` 信息（`prompt_tokens`、`completion_tokens`）。
- 将用量记录写入数据库表 `token_usage`：
  - `id`、`project_id`、`agent_name`、`prompt_tokens`、`completion_tokens`、`model`、`created_at`

**成本计算**：
- 维护一个 `ModelPricing` 配置（可从环境变量或数据库加载）：
  ```json
  {
    "default": {"prompt": 0.002, "completion": 0.006}
  }
  ```
- 在用户仪表盘或项目详情页展示累计费用。

**前端展示**：
- 设置页显示本月累计花费。
- 项目概览显示该项目消耗的 Token 及估算金额。

### 2.4 只读分享链接

**用户价值**：用户可将作品以精美只读页面分享给朋友、编辑或粉丝，无需登录。

**实现方案**：
- 新增表 `share_links`：
  - `id`、`project_id`、`token`（随机唯一字符串）、`expires_at`（可选）、`created_at`
- API：`POST /api/projects/{id}/share` 生成分享链接，返回 `{share_url: "/share/abc123"}`。
- 只读页面 `/share/:token`：
  - 独立于主应用的路由，不要求登录。
  - 渲染项目标题、作者、章节列表，点击章节以优雅排版展示正文（纯阅读模式）。
  - 可包含简单的导航（目录、上一章、下一章）。

### 2.5 简单协作（项目邀请）

**用户价值**：允许作者邀请编辑或合作者共同查看/编辑项目。

**数据模型**：
- 新增表 `project_collaborators`：
  - `id`、`project_id`、`user_id`、`role`（`editor` 或 `viewer`）、`invited_at`、`accepted_at`

**流程**：
1. 项目所有者输入协作者邮箱，系统发送邀请邮件（含邀请链接）。
2. 被邀请者注册/登录后，通过链接接受邀请，成为协作者。
3. 权限控制：
   - `viewer`：只读访问项目所有内容。
   - `editor`：可编辑章节、触发单章重生成，但不能删除项目或管理协作者。
4. 后端中间件需校验当前用户对项目的权限。

**注意**：第一期可简化——仅支持通过用户名添加已注册用户，无需邮件邀请。

## 三、实施顺序建议

| 优先级 | 功能 | 理由 |
|--------|------|------|
| P0 | 多格式导出（至少 `.epub`） | 用户强烈需求，作品“出口” |
| P1 | 版本历史 | 提升写作安全感，低实现成本 |
| P2 | Token 统计 | 为商业化做准备，同时满足用户知情权 |
| P3 | 分享链接 | 增强传播性，低成本获客 |
| P4 | 协作功能 | 复杂功能，可作为 V2.0 亮点 |

## 四、注意事项与风险提示

1. **导出服务性能**：
   - 生成 `.epub` 或 `.docx` 可能耗时（长篇小说），建议也通过 Celery 异步任务执行，返回下载链接时可能需轮询。
   - 临时文件需设置过期清理策略（如 24 小时后删除）。

2. **版本存储膨胀**：
   - 章节内容直接存数据库可能导致表急剧增大。可考虑仅存储差异（diff）或使用对象存储。
   - 初期直接存全量内容，设好保留版本数上限即可。

3. **用量记录准确性**：
   - 必须确保在每次 LLM 调用成功时记录，失败或重试不计。
   - 若使用流式响应，需从最后一个 chunk 获取最终 usage。

4. **分享链接安全**：
   - `token` 使用 `secrets.token_urlsafe(32)` 生成，防猜测。
   - 支持设置有效期，过期后返回 404。

5. **协作权限粒度**：
   - 本期只做项目级，未来可扩展至章节级（如仅共享前几章）。

## 五、验收标准

- [ ] 在项目页点击“导出 EPUB”，稍后收到下载链接，文件可在 Calibre 等阅读器中正常打开。
- [ ] 修改章节内容 3 次后，能在版本历史中看到 3 个版本，并可恢复到第 1 版。
- [ ] 生成 3 章后，设置页或仪表盘能看到 Token 消耗统计和预估费用。
- [ ] 生成分享链接，在无痕窗口中打开能看到只读版本的项目内容。
- [ ] 邀请另一用户协作者，对方登录后能访问被邀请的项目（按角色权限限制操作）。

## 六、结语

完成以上五个 Task 后，StoryForge AI 将具备一个成熟 SaaS 产品的基础骨架：稳定的后端架构、清晰的数据模型、专业的用户界面、以及增强用户粘性的高级功能。后续可在此基础上持续迭代商业化特性（订阅计费、团队空间、Prompt 模板市场等）。
