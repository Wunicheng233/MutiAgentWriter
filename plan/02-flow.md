# StoryForge AI 精简工作流文档

## 1. 架构概览

### 1.1 设计哲学

经过对原 9-Agent 架构的反思，我们发现：
- **Agent 职责重叠**导致修改冲突与误差累积。
- **检查节点过多**导致流程冗长、Token 成本高昂。
- **输出格式不统一**导致系统解析频繁失败。

精简架构遵循以下原则：

| 原则 | 说明 |
| :--- | :--- |
| **LLM 只做不可替代的事** | 能由代码规则完成的检查，绝不调用 LLM。 |
| **一条主线跑通** | 规划 → 写作 → 评审 → 修订，四步闭环。 |
| **输出格式机器友好** | Critic 输出纯 JSON，系统直接解析。 |
| **指令驱动** | Critic 下达可执行指令，Revise 忠实执行。 |

### 1.2 Agent 数量对比

| 版本 | Agent 数量 | 每章 LLM 调用次数 |
| :--- | :--- | :--- |
| 原架构 | 9 个 | 5-7 次 |
| **精简架构** | **4 个** | **2-3 次** |

## 2. Agent 详细定义

### 2.1 Planner — 故事架构师

| 属性 | 值 |
| :--- | :--- |
| **对应 Prompt** | `planner.md` |
| **调用时机** | 项目初始化时，仅调用 **1 次** |
| **输入变量** | `{{content_type}}`、`{{user_requirements}}`、`{{platform}}`、`{{target_words}}`、`{{target_duration}}`、`{{core_hook}}` |
| **输出格式** | Markdown（结构依 `{{content_type}}` 而定） |
| **输出用途** | 作为 `{{world_bible}}` 和 `{{chapter_outline}}` 供后续 Agent 使用 |

**输出内容（以 `novel` 模式为例）**：
- 核心设定（书名、卖点、时代背景及校准）
- 人物档案（主角、配角 ≤3 人）
- 世界规则
- 分章大纲表（每章：本章目标、核心冲突/爽点、结尾钩子）

---

### 2.2 Writer — 叙事作家

| 属性 | 值 |
| :--- | :--- |
| **对应 Prompt** | `writer.md` |
| **调用时机** | 每章生成时调用 **1 次** |
| **输入变量** | `{{world_bible}}`、`{{chapter_outline}}`、`{{previous_summary}}`、`{{target_word_count}}`、`{{content_type}}` |
| **输出格式** | 纯文本正文（第一行：`第X章 [标题]`） |
| **输出用途** | 作为章节初稿，送入系统层防护 |

**核心约束**：
- 严格履约大纲的三个节点（目标、冲突/爽点、钩子）
- 不添加新人物、新设定
- 套话黑名单词汇绝对禁止
- 短段落排版（≤3 句），对话单独成段

---

### 2.3 Critic — 章节评审员

| 属性 | 值 |
| :--- | :--- |
| **对应 Prompt** | `critic.md` |
| **调用时机** | Writer 输出后，Revise 修改后（复评） |
| **输入变量** | `{{chapter_content}}`、`{{world_bible}}`、`{{chapter_outline}}`、`{{content_type}}` |
| **输出格式** | **纯 JSON**（无 Markdown 包裹） |
| **输出用途** | 系统解析 `passed` 决定流程走向；`issues` 交付 Revise 执行 |

**JSON 结构**：
```json
{
  "passed": true,
  "score": 8,
  "issues": []
}
```

**不通过示例**：
```json
{
  "passed": false,
  "score": 5,
  "issues": [
    {
      "type": "结尾乏力",
      "location": "这究竟是怎么回事呢？",
      "fix": "将疑问句结尾改为展示门缝中渗出的诡异光线"
    }
  ]
}
```

**问题类型标签**：`大纲偏离`、`人设崩塌`、`逻辑断层`、`情绪生硬`、`结尾乏力`、`套话过多`、`格式问题`、`时代错位`

---

### 2.4 Revise — 内容修订师

| 属性 | 值 |
| :--- | :--- |
| **对应 Prompt** | `revise.md` |
| **调用时机** | Critic 返回 `passed = false` 时调用 |
| **输入变量** | `{{original_chapter}}`、`{{critic_issues}}`、`{{world_bible}}` |
| **输出格式** | 纯文本正文（修订后完整章节） |
| **输出用途** | 送回 Critic 复评 |

**核心约束**：
- 只修改 `issues` 中指出的位置，不动其他内容
- 严格按 `fix` 字段的指令执行，不自行发挥
- 从后往前处理问题，避免行号偏移
- 保持章节标题、结尾钩子功能不变

## 3. 系统层防护

系统层防护是一套 **纯代码实现的拦截规则**，在 Writer 输出后、Critic 评审前执行。详细定义见《系统层防护规则文档》。

| 检查项 | 严重级别 | 处理方式 |
| :--- | :--- | :--- |
| 章节标题格式 | 阻断 | 自动补全/修正 |
| 敏感词过滤 | 阻断 | 自动替换为 `*` |
| 对话分行 | 自动修正 | 自动拆分 |
| 字数偏离 | 警告 | 记录，继续流程 |
| 段落长度 | 警告 | 记录超标行号 |
| 主角出场 | 警告 | 记录提醒 |
| 结尾钩子 | 警告 | 记录建议 |
| 套话黑名单 | 警告 | 记录列表 |
| 时代错位词 | 警告 | 记录违规词 |

## 4. 占位符替换机制

所有 Prompt 文件中的 `{{variable}}` 占位符由系统在调用 LLM 前进行替换。LLM 只会看到替换后的实际内容。

**占位符总表**：

| 占位符 | 说明 | 使用 Agent |
| :--- | :--- | :--- |
| `{{content_type}}` | `novel` / `short_story` / `script` | Planner, Writer, Critic |
| `{{user_requirements}}` | 用户原始需求文本 | Planner |
| `{{platform}}` | 目标平台（番茄/起点/知乎等） | Planner |
| `{{target_words}}` | 目标总字数 | Planner |
| `{{target_duration}}` | 目标时长/集数（剧本模式） | Planner |
| `{{core_hook}}` | 核心卖点/高概念 | Planner |
| `{{world_bible}}` | Planner 输出的完整设定圣经 | Writer, Critic, Revise |
| `{{chapter_outline}}` | 当前章节的大纲条目 | Writer, Critic |
| `{{previous_summary}}` | 前文摘要（系统生成） | Writer |
| `{{target_word_count}}` | 本章目标字数 | Writer |
| `{{chapter_content}}` | 待评审/修订的章节正文 | Critic, Revise |
| `{{original_chapter}}` | 待修订的原始章节 | Revise |
| `{{critic_issues}}` | Critic 输出的 issues 数组（JSON） | Revise |

## 5. 数据持久化

### 5.1 项目目录结构

```
data/projects/{user_id}/{project_id}/
├── info.json                 # 项目元信息
├── user_requirements.txt     # 用户原始需求
├── setting_bible.md          # Planner 输出的设定圣经
├── chapter_outlines.json     # 分章大纲（结构化存储）
├── chapters/
│   ├── chapter_1.txt         # 第1章正文（终稿）
│   ├── chapter_2.txt
│   └── ...
└── scores/
    ├── chapter_1.json        # Critic 终评分数
    ├── chapter_2.json
    └── ...
```

### 5.2 章节状态追踪

系统需维护每章的状态：

| 状态 | 说明 |
| :--- | :--- |
| `pending` | 尚未开始生成 |
| `drafting` | Writer 正在生成 |
| `reviewing` | Critic 评审中 |
| `revising` | Revise 修订中 |
| `approved` | 已通过评审，定稿 |
| `failed` | 超过最大修订次数，需人工介入 |

## 6. 错误处理与重试

### 6.1 LLM 调用重试

| 场景 | 策略 |
| :--- | :--- |
| API 超时/网络错误 | 最多重试 3 次，指数退避 |
| 输出格式解析失败 | 调用 Fix 逻辑（代码层正则提取/修复），不重试 LLM |
| Critic 输出非 JSON | 尝试正则提取 JSON，失败则返回默认 `passed=false` 并记录错误 |

### 6.2 修订循环控制

| 参数 | 默认值 | 说明 |
| :--- | :--- | :--- |
| `max_revise_loops` | 2 | 单章最多 Critic → Revise 循环次数 |
| 超过上限后 | 章节标记为 `failed`，通知用户介入 |

| 指标 | 原架构（估算） | 精简架构（估算） |
| :--- | :--- | :--- |
| 每章 LLM 调用次数 | 5-7 次 | 2-3 次 |

## 8. 扩展与定制

### 8.1 新增创作模式

1. 在 `planner.md` 的模式切换表中添加新模式。
2. 添加对应的输出模板章节。
3. 更新 `{{content_type}}` 的枚举值。

### 8.2 调整评审严格度

- 修改 `critic.md` 中的 **通过标准** 描述。
- 调整系统层防护中的警告阈值（如字数偏差比例）。

### 8.3 自定义套话黑名单

- 修改 `guardrails_config.yaml` 中的 `cliche_blacklist` 列表。
- 系统层防护会自动加载新配置。

---

## 9. 总结

精简架构将原有 9 个 Agent 压缩为 4 个核心 Agent，并将基础检查剥离为系统层代码规则。这套架构：

- **职责清晰**：Planner（规划）、Writer（执行）、Critic（判定）、Revise（修正）
- **格式统一**：Critic 输出纯 JSON，系统稳定解析
- **成本可控**：每章 LLM 调用次数降低 60% 以上
- **易于维护**：Prompt 文件独立，占位符机制清晰

结合已完成的四份 Agent Prompt 和系统层防护文档，开发团队可直接据此进行编码实现。
