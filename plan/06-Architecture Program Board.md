## 2. 现实基线

在继续推进之前，我们必须先承认当前系统的真实形态。

### 2.1 当前最接近的系统定义

StoryForge AI 当前不是一个成熟的“多智能体协作平台”，而是一个：

`带有多角色分工意识的、可暂停/恢复的、章节级小说生成流水线原型`

它已经具备以下非常重要的基础：

- 有前后端、数据库、异步任务、编辑器、分享和导出闭环
- 已经有 Planner / Writer / Critic / Revise 的最小 Agent 分工
- 已经开始出现 workflow、artifact、feedback 的数据库抽象
- 已经开始引入人工确认，而不是纯黑盒自动生成

但它还没有真正做到以下事情：

- workflow 成为唯一运行时事实源
- agent 间通信具备强契约
- 项目隔离不依赖全局运行时上下文
- harness 可以稳定回答“这次优化是否真的更好”
- 文档、规划、实现三者保持长期一致

### 2.2 当前最危险的系统级缺口

结合现有代码和深度研究报告，当前最值得优先处理的不是“功能数量不够”，而是下面五类结构性问题：

1. `事实源分裂`
   数据库、文件系统、Celery 状态、向量索引并存，恢复与回放容易出现分裂状态。

2. `运行时上下文全局化`
   项目隔离、检索命名空间、世界观状态仍然部分依赖全局状态，未来扩并发和多项目运行会很脆弱。

3. `工作流契约不足`
   当前很多步骤仍然更接近脚本过程，而不是标准化 step run / artifact run。

4. `评测与可观测空心化`
   有打分与 guardrails，但缺少真正可回归的 harness、指标面板与失败归因体系。

5. `文档与实现分叉`
   README、架构描述、plan 文档和真实运行路径之间，仍有历史演进造成的认知偏差。

## 3. 不可动摇的推进原则

后续所有技术规划都要服从下面五条原则。

### 3.1 数据库是真相，文件系统是派生层

长期目标必须非常明确：

- 数据库保存业务状态与流程状态
- 文件系统保存快照、调试产物、导出内容和兼容层
- 向量索引是可重建派生层
- Celery state 是执行期元数据，不是业务真相

任何新功能如果继续让文件承担主事实源，都会把 split-brain 问题继续放大。

### 3.2 先强化合同，再增加 Agent

未来真正限制系统上限的，不是 Agent 数量，而是 Agent contract 是否稳定。

所以优先级必须是：

1. 定义输入输出 schema
2. 定义 artifact 类型
3. 定义 step 生命周期
4. 定义失败类型和恢复策略
5. 在此基础上再扩展新的质量 Agent 或结构化提取 Agent

### 3.3 先做模块化单体，再谈分布式

现阶段最优解仍然是：

- 单仓库
- 单后端服务
- Celery 负责任务边界
- 数据库承担持久化事实源

不要过早拆服务。真正先要拆清楚的是模块边界、上下文边界和状态边界。

### 3.4 工作流优先于 prompt 微调

如果 workflow、artifact、feedback、replay 都不稳定，那么 prompt 调优只会制造更多不可解释差异。

所以长期收益排序应当固定为：

1. workflow 可控
2. artifact 可追踪
3. feedback 可结构化
4. harness 可回归
5. 然后才是 prompt 与模型精修

### 3.5 计划必须领先三步，但施工只推进一步

后续每一次正式开发，都遵循这个节奏：

- 规划层：至少看到后三步
- 实施层：只做当前一步
- 验证层：当前一步稳定后再进入下一步

这意味着路线图不能停，但代码施工必须节制，避免把系统拖回“边修边乱”的状态。

## 4. 风险总账

下表是当前阶段最需要长期跟踪的技术风险总账。它不是一次性 checklist，而是后续每一轮推进都要回看的约束表。

| 风险 | 现在的表现 | 如果继续放大 | 优先级 | 应对阶段 |
|---|---|---|---|---|
| 状态分裂 | 数据库、文件、任务状态和索引层各自保存部分真相 | 恢复失败、章节错乱、导出错版本、人工反馈被吞 | P0 | Wave A / Wave B |
| 全局上下文污染 | 项目路径、检索命名空间、世界观状态部分依赖全局运行时上下文 | 多项目并发时串数据，扩 worker 后事故概率激增 | P0 | Wave B |
| 工作流只是脚本 | 流程可跑，但 step 输入输出和状态机不够显式 | 无法优雅支持重放、回退、分叉、单章重跑 | P0 | Wave A / Wave B |
| feedback 不是一等对象 | 当前仍保留较重的文件兼容逻辑 | 人工协作、社区评论反哺、反馈分析都无法成立 | P0 | Wave A |
| harness 空白 | 调优主要依赖主观判断 | 优化方向无法证伪，越做越重也未必更好 | P1 | Wave C |
| agent 契约弱 | 很多节点仍依赖文本约定与隐式上下文 | 换模型、换 prompt、插新节点时稳定性下降 | P1 | Wave B / Wave C |
| 文档分叉 | README、plan、代码现实存在偏差 | 新人接手成本高，外部沟通与内部判断失真 | P1 | Wave A |
| 安全边界薄弱 | 密钥、认证日志、分享权限等存在历史设计债务 | 线上事故与合规风险 | P1 | Wave A |
| 测试断层 | 核心链路回归还不够系统化 | 每一轮重构都在重复踩坑 | P1 | Wave A / Wave C |

## 5. 分波次推进图

后续 4 到 6 个月，技术路线建议拆成四个连续波次，而不是很多并行主题。

### Wave A：状态收束与兼容层治理

时间建议：

- 当前到未来 3 周

目标：

- 让系统从“能跑”进入“状态不再继续发散”

核心工作：

- 完成 `run / step / artifact / feedback` 最小闭环
- 清理仍然依赖文件主导判断的关键路径
- 把 `waiting_confirm`、重写、继续生成、恢复执行的状态收敛到统一集合
- 建立当前事实源地图：
  - 哪些状态在 DB
  - 哪些状态在文件
  - 哪些是兼容桥接，不再允许新增依赖
- 明确 README 与计划文档中的“当前实现”描述

这一波的成功标准：

- 核心创作链路恢复时不再优先依赖“文件是否存在”
- 用户反馈不会被静默跳过
- 当前 workflow run 能解释任务走到了哪一步
- 兼容文件只作为桥接层存在，而不是继续扩散

这一波坚决不做：

- 不扩新的复杂 Agent 角色
- 不做场景级 DAG
- 不做社区化功能

### Wave B：运行时去全局化与 Agent Contract v1

时间建议：

- 未来第 3 到 8 周

目标：

- 让项目隔离与节点执行真正建立在显式上下文之上

核心工作：

- 去掉核心运行链路对全局项目上下文的依赖
- 把 `project_id / project_path / namespace / run_id` 显式穿透到：
  - orchestrator
  - retrieval
  - worldview / memory
  - artifact store
- 为核心 Agent 定义注册信息与最小 contract：
  - input schema
  - output schema
  - prompt version
  - model policy
  - retry policy
  - fallback policy
- 让主要步骤具备标准错误类型，而不是大量自由文本异常

这一波的成功标准：

- 单次运行的关键上下文可以完整追踪
- 主要 Agent 都有统一注册元数据
- 后续增加新的质量节点时，不需要再靠散落字符串接线

这一波坚决不做：

- 不做微服务拆分
- 不追求“更多 agent 看起来更高级”

### Wave C：Harness 与可观测基线

时间建议：

- 未来第 2 到 4 个月

目标：

- 让系统进入“可验证优化”阶段

核心工作：

- 建立 harness v1：
  - golden set
  - failure set
  - fixed run config
  - regression report
- 建立 step 级指标采集：
  - latency
  - token usage
  - success rate
  - rewrite rate
  - user rejection rate
- 建立 artifact 与评分的可追踪关系
- 让 prompt / workflow / model 变更具备最小 A/B 比较能力

这一波的成功标准：

- 每次工作流或 prompt 调整都能被离线回归
- 团队可以回答“这次为什么变好/变差”
- 不再只靠主观阅读判断优化效果

这一波坚决不做：

- 不做大规模推荐系统
- 不把 observability 搞成重型平台工程

### Wave D：作者工作台与可回放创作

时间建议：

- 未来第 4 到 6 个月

目标：

- 把创作系统从生成流水线升级为持续创作工作台

核心工作：

- 章节版本树
- artifact 差异对比
- 明确的“重写当前章 / 从某版本分叉 / 回退到指定版本”
- 结构化记忆面板：
  - 人物卡
  - 时间线
  - 设定圣经
  - 伏笔清单
- 让用户反馈、系统评审、未来评论输入都能汇入统一反馈模型

这一波的成功标准：

- 创作不再只是“一次生成，一次查看”
- 用户可以持续在同一项目中编辑、比较、回放和分叉
- 为后续社区与发布生态打下技术底座

这一波坚决不做：

- 不急着做复杂商业化逻辑
- 不急着做推荐流和作者成长体系

## 6. 四条主施工线

为了避免“表面上按阶段推进，实际上所有事情还是糊在一起”，后续具体工作要被组织为四条主施工线。

### 6.1 Workflow 线

负责回答：

- 当前 run 是什么
- step 走到哪里
- 从哪里恢复
- 为什么被打回
- 这次重写和上一次有什么关系

长期产物：

- `WorkflowRun`
- `WorkflowStepRun`
- `checkpoint`
- `approval event`
- `retry / replay / skip`

### 6.2 Artifact & Data 线

负责回答：

- 当前计划稿、设定稿、章节稿、修订稿、导出稿分别是什么
- 谁是当前版本
- 哪些只是派生文件
- 哪些数据可以重建

长期产物：

- artifact taxonomy
- version graph
- derived file policy
- retrieval rebuild policy

### 6.3 Agent Runtime 线

负责回答：

- Agent 接收什么输入
- 输出什么结构
- 用什么模型
- 出错怎么分类
- 重试和 fallback 是什么

长期产物：

- agent registry
- schema contract
- model policy
- prompt version policy

### 6.4 Harness & Observability 线

负责回答：

- 这次改动值不值得
- 哪一步最容易失败
- 成本花在哪里
- 用户为什么拒稿
- 哪类 prompt 变更真正有效

长期产物：

- benchmark set
- regression runner
- score report
- failure taxonomy
- step metrics dashboard

## 7. 下一阶段最值得真的开工的 12 个主题

这 12 个主题不是一次全做，而是接下来连续几个迭代最值得推进的技术主题池。

### 7.1 Wave A 主题池

1. `事实源地图`
   把项目、章节、任务、artifact、反馈、导出、索引的真相位置列清楚。

2. `反馈闭环收束`
   让 feedback 入库、消费、标记已应用形成闭环，文件只做兼容桥接。

3. `活跃任务状态统一`
   所有生成、导出、重置入口共用同一套 active task status 集合。

4. `安全与权限快速止血`
   清理敏感日志、收紧分享和导出授权、校正 API key 表达方式。

### 7.2 Wave B 主题池

5. `运行时上下文对象`
   定义 `RunContext` 或等价对象，把项目隔离信息显式传透。

6. `检索去全局化`
   让 retrieval namespace 与 artifact scope 绑定到 run/project，而不是全局可变状态。

7. `Agent Registry v1`
   核心 Agent 的注册元数据与执行契约落地。

8. `结构化错误类型`
   统一 step failure reason，减少自由文本异常支配系统分支。

### 7.3 Wave C 主题池

9. `Harness v1`
   跑固定输入、固定配置、固定评分脚本。

10. `Step Metrics`
   记录时延、token、失败率、重写率、拒稿率。

### 7.4 Wave D 主题池

11. `章节版本树`
   支持历史版本、AI 修订版、人工版、恢复点。

12. `结构化记忆面板`
   让人物、时间线、设定、伏笔从隐式文本变成系统可读对象。

## 8. 顺序依赖

很多事情听起来都重要，但不是所有事情都能并行开。

后续必须遵守下面这条依赖链：

`状态收束 -> 上下文去全局化 -> Agent contract -> Harness -> Workspace / Branching -> 社区生态`

这条依赖的意义很大：

- 如果状态还没收束，就不要急着做复杂版本树
- 如果上下文还是全局的，就不要急着提并发和多 worker
- 如果 Agent contract 还没稳定，就不要急着横向加很多专用 Agent
- 如果 harness 还没有，就不要过度相信 prompt 调优结果
- 如果 workspace 还不稳，就不要急着做社区反哺创作

## 9. Definition of Done

未来每个阶段都很容易“看起来做了很多”，但实际上只是新增了更多复杂度。

所以这里给出一组统一的完成定义。

### 9.1 Wave A 完成定义

- 核心 workflow 状态和反馈状态在数据库中可解释
- 文件兼容层不再扩张
- 核心入口的 active task 判断统一
- 当前实现的系统描述与文档一致

### 9.2 Wave B 完成定义

- 关键运行链路不再依赖全局项目上下文
- retrieval / memory / artifact 的上下文边界明确
- 核心 Agent 有可检查的注册元数据和 schema 契约

### 9.3 Wave C 完成定义

- 至少有一套固定样本可自动回归
- prompt / workflow / model 变更可输出比较报告
- 核心 step 的失败和成本可以被观测

### 9.4 Wave D 完成定义

- 章节可以被版本化、回放、分叉和恢复
- 作者能看见差异、反馈来源和重写路径
- 创作系统具备支撑社区反哺的结构化数据基础

---

## 10. 结论

从长期项目的角度看，StoryForge AI 现在最需要的不是“把 demo 再堆得更复杂”，而是把创作系统真正变成：

- `workflow first`
- `artifact native`
- `human-in-the-loop recoverable`
- `context isolated`
- `evaluation driven`

只要这五件事逐步成立，后面的作者工作台、小说社区、评论反哺、打赏、发布生态，都会变成顺势长出来的能力。

如果这五件事不先成立，那么后面无论加多少 Agent、做多少页面、做多少论坛功能，都会建立在一个不稳定的底座上。
